## Scope of this slice

This protocol defines the fixture, condition, adapter, repetition, ordering, deterministic workspace materialization, pre-result Adaptive selection, duplicate-safe fake-runtime execution, normalized execution-evidence boundary, and the Issue #204 evaluator isolation checkpoint used by the redesigned benchmark portfolio. Normalization may classify committed terminal status and telemetry availability, and evaluator envelopes may record raw typed observations, but neither layer scores or compares conditions, authorizes measured execution, freezes product-value thresholds or weights, or publishes a product recommendation.

No cross-condition outcome is inspected or generated by the `plan` command. Public answer content from Issues #193 through #196 is contaminated and cannot be used as evaluator source; clean fixture and private evaluator work remains in #206 through #209.

It is a deterministic metadata-level duplicate diagnostic, not a proof of semantic identity or task independence. It uses no LLM, embedding, network input, timestamp, or execution result.

## Deterministic materialization

`materialize` consumes a previously validated execution plan. It re-derives config, protocol, repository, seed, plan, case, block, repetition, ordering, adapter, and fixture-manifest identity before creating a case. It does not generate a seed or silently replace the supplied plan.

The output must be absent or empty and must not traverse a symlink. Materialization uses a sibling staging directory, removes incomplete staging output on failure, and atomically renames the completed run only after the versioned manifest passes JSON Schema validation. Existing materialized runs are never overwritten.

Fixture manifests are the explicit agent-visible allowlist. Only `task.md` and declared `workspace/**` regular files are copied; path escape, absolute paths, resolved symlinks, non-agent-visible classification, evaluator directories, expected outputs, hidden tests, oracle/reference/rubric/scoring assets, and prior condition results fail closed. `BENCHMARK_TASK.md` preserves the exact task bytes, and workspace files preserve bytes and executable modes.

The materialization manifest records agent-visible inventories separately from condition projections. For every adapter, fixture, and repetition block, all four conditions must share the same task, workspace, and frozen-input digests.

Condition projections are independent:

- Plain has no ASK assets.
- Kernel-only has only canonical `AGENTS.md`.
- Adaptive ASK has canonical Kernel plus an adapter-owned boundary recording that mechanisms are not selected, a selection seal is not produced, and runtime execution is not attempted.
- Full ASK uses the current repository-local full installer projection for the selected adapter.

Codex assets stay under Codex-owned paths and Claude assets stay under Claude-owned paths. Repository projection evidence does not establish runtime loading or behavior; the manifest records that downgrade instead of simulating runtime capability.

## Conditions

| Condition | Product role | Instruction surface |
|---|---|---|
| `plain` | context baseline | fixture context without ASK |
| `kernel_only` | ordinary-work default | canonical Kernel only |
| `adaptive_ask` | primary product hypothesis | Kernel plus mechanisms selected from observed evidence before the result |
| `full_ask` | diagnostic maximum | full selected adapter profile |

Full ASK is not a default-product condition. Lightweight bypass is valid for Adaptive ASK and remains distinct from under-processing.

## Adapter tracks

Codex and Claude are separate tracks. The plan contains an explicit `adapter_track` for every case, and `pool_adapter_results` is false. A runtime path that is not evidenced later must be recorded as unavailable rather than scored as zero or simulated.

## Deterministic balanced ordering

`plan` creates one four-condition block for each adapter, fixture, and repetition. A seed-derived base permutation is rotated once per repetition. This gives every condition a unique position in each block and keeps position-count imbalance at no more than one for the registered three- and five-repetition schedules.

The operator supplies a non-sensitive canonical seed. The durable plan stores its `seed_id`, public value, and SHA-256 digest so another operator can regenerate the plan from the artifact alone. Automatic seed generation is not supported by this foundation slice.

Every plan has a namespace:

```text
plan_id = SHA-256(config digest, protocol digest, repository revision, canonical seed)
```

The plan digest is embedded in every case and block identifier. A config, protocol, repository, or seed change therefore creates a different resume namespace instead of reusing prior case identities.

The portfolio config and emitted plan are validated as complete instances against their declared JSON Schemas through the repository's shared validator. Validation happens before a plan is written. Adaptive selection records use the same shared validator entrypoint.

## Adaptive pre-result selection and seal

Only an existing Adaptive case may be sealed. `seal-selection` consumes a previously validated plan and materialization; it never creates a plan, rematerializes a case, invokes an adapter, or writes into an agent-visible case directory.

```bash
node scripts/ask-benchmark.mjs seal-selection \
  --config benchmarks/adaptive-portfolio.config.json \
  --plan /path/to/execution-plan.json \
  --materialized /path/to/materialized-root \
  --state-dir /path/to/non-agent-visible-state \
  --case-id case-... \
  --input /path/to/selection-input.json

node scripts/ask-benchmark.mjs verify-selection \
  --config benchmarks/adaptive-portfolio.config.json \
  --plan /path/to/execution-plan.json \
  --materialized /path/to/materialized-root \
  --state-dir /path/to/non-agent-visible-state \
  --case-id case-...
```

Before accepting a seal, the consumer revalidates the plan and manifest Schema plus plan/config/protocol/repository/materializer identities, case count and namespaces, all plan-to-case fields, four-condition block equality, projection adapter/profile/fingerprint, actual inventories, SHA-256 digests, modes, path/symlink gates, and evaluator exclusion. It re-anchors every frozen inventory to the plan-pinned fixture input manifest, mapping fixture `task.md` to materialized `BENCHMARK_TASK.md`, so a matching rewritten materialization manifest cannot replace fixture bytes. It does not trust manifest status strings without checking the current case bytes.

The normalized selection input records task class and at least one non-blank observed signal; selected and skipped mechanisms; required gates; requested and omitted agents; expected evidence; capability downgrades; lightweight-bypass use and rationale; and the exact adapter/profile/renderer/projection fingerprint. The sealed record additionally binds `plan_id`, canonical plan digest, materialization-manifest digest, output-root identity, materializer version/revision, case/block/adapter/condition/fixture/repetition identity, frozen-input digest, condition-projection digest, and projection fingerprint. It has no result, score, correctness, recommendation, completion, hidden-test, oracle, or evaluator fields.

The selection timestamp is RFC 3339 and is included in the sealed content. The CLI always creates it from its clock; the command surface has no timestamp override. Focused tests inject a clock only through the module boundary. A caller-supplied digest is rejected. `selection_digest` is SHA-256 of the exact canonical JSON record with only `selection_digest` omitted: object keys sort lexicographically at every nesting level, JSON primitive spelling is preserved, and array order remains semantic. Schema-declared unique arrays reject duplicates. This prevents self-hashing and makes property insertion order irrelevant; changing any bound field changes the digest.

State is outside every case root at `<state-dir>/selections/<case-id>.json`. A versioned `<state-dir>/selection-state.json` binds the state root to one plan and materialization and records prior seals, so deleting a selection file cannot permit recreation. State-root and selection paths reject regular-file roots and symlink traversal; unknown state entries, state under a case, overwrites, and foreign plan/materialization identities fail closed. Publication stages files under the state root, creates the seal without replacement, updates the index atomically, removes staging on failure, and makes sealed files read-only where supported. This is deterministic tamper evidence and workflow control, not cryptographic protection against an administrator rewriting the entire filesystem.

Lightweight bypass is valid only with a non-empty reason, no selected mechanisms, and an explicit non-empty skipped set; an empty ordinary selection is not a bypass. Capability downgrades explicitly preserve capability and reason, cannot also be listed as a selected mechanism, and do not substitute the other adapter or score an unavailable runtime as zero. The selection boundary records a decision; it does not claim that any mechanism or runtime was executed.

Selection rejects Plain, Kernel-only, Full ASK, missing/stale/foreign cases, an already sealed case, and any benchmark artifact at every case path, including `workspace/**/.benchmark-*`. Generic result/output/score/telemetry names remain workspace-exempt so ordinary fixture source is not rejected merely by its filename. The same result-artifact helper is the required integration point for a later runtime slice. Both selection commands require tracked working-tree and index bytes to match `HEAD` before reading config, schemas, plan, or state. `verify-selection` performs no writes and exits nonzero for missing or invalid state.

## Runtime execution and partial resume

`execute-portfolio` consumes an existing plan, materialized root, complete sealed-selection state, separate external run root, one adapter runtime config, and one executable path. It revalidates the tracked tree/index, plan/config/protocol/repository identities, materialization consumer contract, fixture re-anchoring, actual bytes/modes/digests, adapter/condition projections, evaluator/path/symlink gates, selection-state identity, and output Schema before it can spawn an executor.

```bash
node scripts/ask-benchmark.mjs execute-portfolio \
  --config benchmarks/adaptive-portfolio.config.json \
  --plan /path/to/execution-plan.json \
  --materialized /path/to/materialized-root \
  --selection-state /path/to/non-agent-visible-selection-state \
  --run-dir /path/to/non-agent-visible-run-state \
  --adapter codex \
  --runtime-config /path/to/codex-runtime.json \
  --agent-bin /path/to/executable

node scripts/ask-benchmark.mjs verify-execution \
  --config benchmarks/adaptive-portfolio.config.json \
  --plan /path/to/execution-plan.json \
  --materialized /path/to/materialized-root \
  --selection-state /path/to/non-agent-visible-selection-state \
  --run-dir /path/to/non-agent-visible-run-state
```

The run root records immutable `run-identity.json`, one compare-and-set runtime identity per adapter, and independently published per-case state/attempt files. It is outside both materialized and selection roots. Each case writes a complete claim into a sibling staging directory and atomically renames that directory to `claim`; an incomplete staging claim is never treated as active. Terminal publication writes a request-bound result, approved final JSON when completed, and `commit.json` digest manifest before deriving state and releasing the exact claim. Re-running recovery reconciles a valid terminal result without replacing it or appending an attempt. Completed cases are Schema/digest revalidated and skipped on resume; failed cases require `--retry-failed`; active and expired claims are never stolen automatically. An expired claim or incomplete staging claim can only be reconciled by its exact claim ID and an operator-supplied reason:

```bash
node scripts/ask-benchmark.mjs recover-case \
  --run-dir /path/to/non-agent-visible-run-state \
  --case-id case-... \
  --claim-id claim-... \
  --reason "operator confirmed worker exit"
```

Every attempt copies only declared materialized case inputs into a tokenized OS temporary workspace outside the durable run root; the materialized root is never a working directory. Success, failure, and timeout remove that workspace in `finally`; exact stale-claim recovery removes a workspace left by hard interruption. Adaptive execution verifies its seal before projection, before process spawn, and after process exit. It projects only sealed selected mechanisms and required gates with the adapter's canonical projection-plan closure; a lightweight bypass projects no additional Skills, and no Adaptive case can fall back to Full ASK. Codex records and executes the same model, reasoning, sandbox, approval, ephemeral, output-Schema, and user-config/rules-isolation command. Claude runs only when its placeholder counts, policy arguments, command flags, `--help`, and `--version` probes confirm the effective non-interactive contract; otherwise it is recorded as `unavailable` without spawning a substitute executor.

Every terminal attempt requires a closed request/result/commit chain bound to the plan, materialization, frozen input, case, attempt, adapter, condition, runtime identity, effective command, and Adaptive selection when applicable. Completed attempts additionally require exit code zero and an `agent-output.schema.json`-valid final JSON matching its digest and byte count. Durable attempts contain only `request.json`, `result.json`, `commit.json`, and completed `final.json`; they never store raw prompts, raw stdout/stderr, full event streams, transcripts, complete source copies, secrets, customer data, or personal data. `verify-execution` is read-only, validates every terminal status, and classifies every case as pending, active, completed, failed, unavailable, interrupted, or invalid.

## Normalized execution evidence

`normalize-execution` treats the run, plan, materialized root, and selection state as read-only inputs. Before deriving output it invokes the same canonical execution consumer used by `verify-execution`, including plan/materialization/selection revalidation, request/result/commit/final digest closure, runtime/environment identity, terminal-attempt continuity, adapter/case/run separation, symlink/path boundaries, and exact managed inventories. A corrupt source, incomplete staging artifact, or unexpected attempt prevents publication; a valid terminal `invalid` outcome remains a normalized outcome rather than becoming a score.

Normalized attempt records use `normalized-portfolio-result.schema.json` and bind the run instance, plan, repository, materialization, fixture input, case, attempt, adapter, condition, repetition, order/block, runtime, effective command, environment snapshot, request, raw result, terminal commit, final output, and Adaptive selection identities. IDs and digests are re-derived from verified evidence. Completed, failed, unavailable, interrupted, and invalid attempts use the execution outcome directly. Every committed attempt is published even while a later retry is active; the uncommitted active attempt is never fabricated, and the latest attempt never hides earlier evidence.

The output is a run-specific collection owned by `normalized-results-root.json`. Each immutable source snapshot is separated under `generations/snapshot-<digest>/`, and each generation separates attempt records under `adapters/codex/` and `adapters/claude/`; the generation manifest fixes `pool_adapter_results` to false. `normalized-run.json` uses `normalized-portfolio-run.schema.json`, binds case states, committed attempts, and adapter identities through `source_snapshot_digest`, and reports completeness by case, adapter, condition, status, and telemetry field. `normalized_run_digest` covers the complete manifest except that self-digest field. Pending and active cases make the manifest partial and appear in `missing_case_ids`; any earlier committed attempts still contribute to `normalized_attempts` and telemetry coverage. Coverage counts are evidence availability, not quality or value.

Every telemetry field has a typed status. Values proven by committed runtime artifacts are `known`; evidence an available runtime did not produce is `unknown`; evidence impossible because the runtime was unavailable is `unavailable`; evaluator-only or terminal-inapplicable fields are `not_applicable`. `harness_spawned_secondary_agent_count` is limited to agents started by the harness. `runtime_agent_count` and `subagent_activity` are `unknown` when an available runtime omits them and `unavailable` when the runtime itself is unavailable; harness zero is never reused as runtime zero. Token, cost, tool/file access, human-effort, unsafe-action, runtime-agent, subagent-activity, and evaluator-quality values are never inferred or converted to zero.

Initial collection publication and every generation publication use sibling staging plus an atomic directory rename. An absent or empty target may be claimed; a new snapshot for the same run appends a new immutable generation; identical generation bytes are idempotent; unmanaged, cross-run, conflicting, symlinked, or overlapping-input output is rejected without replacement. The collection root, `generations/`, every generation, and `normalized-results-root.json` itself must be non-symlink filesystem entries of the expected type. Abandoned staging is detected and requires explicit inspection. The default `verify-normalized-results` re-derives the current source snapshot and expected byte set from authoritative execution artifacts. `verify-normalized-results --snapshot-digest <digest>` instead validates the selected generation's schemas, identities, inventory, lineage closure, coverage, publication digest, and manifest digest without consulting the current execution source; success means self-consistent, not latest. Both modes are read-only.

The historical B/B2/C `result.schema.json` remains a separate scored-result contract. It is not silently treated as normalized execution evidence: supplying a historical result file to `normalize-execution` is deterministically rejected as unsupported and migration-required. A future migration must be separately versioned and approved.

Normalized execution evidence is a derived artifact, not an evaluator result, score, mechanism scorecard, practice-weighted value, variance conclusion, or product-value claim. Raw execution artifacts remain authoritative. Measured execution is not approved and #198 Stage 0 remains blocked. The Issue #204 boundary checkpoint below does not create the #206–#209 clean fixtures/private packages or complete Issue #204.

## Evaluator isolation boundary checkpoint

The public evaluator reference, private evaluator bundle manifest, and evaluator result envelope are versioned by `evaluator-reference.schema.json`, `private-evaluator-bundle.schema.json`, and `evaluator-result-envelope.schema.json`. All three are closed Schemas. Public artifacts retain only bounded IDs, digests, typed states, categories, severities, byte counts, and evidence references; they cannot contain oracle, rubric, hidden-test, matcher, reference-answer, expected-patch, raw evaluator-prompt, private-path, credential, secret, customer-data, or personal-data fields.

The private root must be physically outside the repository and disjoint from materialized, selection-state, execution-run, normalized-results, and staged public-artifact roots. Every supplied boundary root must exist as a real directory; missing paths, regular files, root symlinks, and canonical private/public overlap are rejected. Managed materialization, selection-state, run, and normalized marker files are required, and full result verification binds their file or canonical identity to the verified normalized lineage. The manifest uses a deterministically ordered exact inventory of portable relative regular files. Duplicate roles or paths, absolute POSIX paths, Windows drive/UNC/device paths, backslashes, path escape, missing files, unexpected files, digest or byte drift, and semantic manifest modification fail closed.

The bundle ID binds Schema version/path, fixture identity, fixture-input digest, and complete asset inventory. The bundle digest covers the full manifest except its own digest field, thereby closing over the derived bundle ID, all asset hashes and byte counts, generator identity, independence statement, review provenance, capabilities, and boundary declarations. The public metadata digest similarly covers the complete public reference except itself. A reference and bundle must agree on bundle, fixture, input, task, suite, revision, generator, independence, and review identities, preventing cross-fixture or cross-input reuse without detectable identity change.

Evaluator-result verification is read-only and invokes the existing normalized-results immutable-snapshot verifier. It resolves the referenced normalized attempt only after that verifier has checked the collection Schema, root/generation identities, exact inventory, per-result identities, snapshot closure, case/attempt/adapter lineage, telemetry coverage, publication digest, and normalized manifest digest. The evaluator envelope then binds normalized result, run, plan, fixture/input, case, attempt, adapter, condition, repetition, source snapshot, evaluator bundle, and evaluator revision. Cross-run, cross-case, cross-attempt, cross-adapter, normalized-result, fixture, input, bundle, or evaluator-result drift is rejected.

The evaluator result contains raw observations only. Metric meaning, equivalent-solution rules, false-positive boundaries, ceilings, floors, evidence-removal gates, weights, thresholds, sensitivity ranges, and aggregation eligibility are owned by #205. Deterministic joining and scoring are owned by #197. The 24 private evaluator packages and clean fixture work are not part of this checkpoint, Issue #204 remains open, and #198 Stage 0 remains blocked.

Private evaluator packages must never be committed to the public repository or uploaded by public CI. Bounded streaming SHA-256 inspection rejects byte-identical private manifest or asset bytes, including copies and hard links, in every public-side boundary root and the Git-managed repository inventory. This exact-byte guarantee does not cover partial extraction, re-encoding, transformation, or semantic equivalence. Full `verify-evaluator-boundary` verification requires a staged public-artifact root; bundle/result-only commands state when publication was not verified. The measured agent must also be isolated from contaminated GitHub Issue/PR content and from evaluator access through web or connectors. See `benchmarks/evaluator-boundary.md` for the exact limits, closure, and CLI contract.

## Privacy and stop boundary

Raw prompts, full outputs, full event streams, full source, secrets, customer data, and personal data remain outside durable plans, selections, and normalized results. Selection state is non-agent-visible and must not be copied into `workspace/` or exposed through `BENCHMARK_TASK.md`. Canonical randomization seeds are public preregistration inputs and must not contain sensitive data. Focused tests may invoke fake Codex/Claude executables only. Measured Codex/Claude execution, scoring, comparative telemetry analysis, evaluator/oracle inspection, condition-result interpretation, and product recommendations remain unauthorized until clean fixture/evaluator dependencies are complete and the required manifests, evaluator digests, thresholds, weights, runtime variables, randomization seeds, and human-evaluation instructions are frozen under #198.
