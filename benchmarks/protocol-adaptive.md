# Adaptive ASK Portfolio Foundation Protocol

Protocol version: `3.0.0-foundation`
Protocol status: foundation contract; not frozen for measured execution
Parent issue: #197
Portfolio parent: #192
Dependencies: #179 and Draft PR #191

## Scope of this foundation

This protocol defines the fixture, condition, adapter, repetition, ordering, and pre-result Adaptive selection contracts used by the redesigned benchmark portfolio. It does not authorize measured execution, freeze product-value thresholds or weights, or publish a product recommendation.

No cross-condition outcome is inspected or generated by the `plan` command. Fixture authors in #193–#196 can consume the contract only after this slice is reviewed.

## Conditions

| Condition | Product role | Instruction surface |
|---|---|---|
| `plain` | context baseline | fixture context without ASK |
| `kernel_only` | ordinary-work default | canonical Kernel only |
| `adaptive_ask` | primary product hypothesis | Kernel plus mechanisms selected from observed evidence before the result |
| `full_ask` | diagnostic maximum | full selected adapter profile |

Full ASK is not a default-product condition. Lightweight bypass is valid for Adaptive ASK and remains distinct from under-processing.

## Adapter tracks

Codex and Claude are separate tracks. The plan contains an explicit `adapter_track` for every case, and `pool_adapter_results` is false. A runtime path that is not evidenced later must be recorded as unavailable rather than scored as zero or simulated.

## Calibration registration

Their agent-visible bytes remain pinned by `benchmarks/fixtures/checkpoint-b2/input-manifest.json`. Evaluator directories are not execution inputs.

## Deterministic balanced ordering

`plan` creates one four-condition block for each adapter, fixture, and repetition. A seed-derived base permutation is rotated once per repetition. This gives every condition a unique position in each block and keeps position-count imbalance at no more than one for the registered three- and five-repetition schedules.

The durable plan stores only the seed digest, never the raw seed. Case and block identifiers include the adapter track and are deterministic for the same repository revision, config, and seed.

## Adaptive pre-result selection

Every Adaptive run must eventually produce a separate record conforming to `benchmarks/schemas/adaptive-selection.schema.json` before task outcome scoring. It records:

- task class and observed signals;
- selected and skipped mechanisms;
- required gates;
- agents requested and omitted;
- expected evidence;
- capability downgrades;
- lightweight-bypass use and rationale;
- adapter/profile/renderer/projection fingerprint;
- selection timestamp and SHA-256 digest.

The selection record has no result, score, correctness, recommendation, or completion fields. A later runner slice must seal its digest before outcome generation and reject post-result mutation.

## Privacy and stop boundary

Raw prompts, full outputs, full event streams, full source, secrets, customer data, and personal data remain outside durable plans and results. Stop before measured execution until fixture manifests, evaluator digests, thresholds, weights, runtime variables, randomization seeds, and human-evaluation instructions are frozen under #198.
